# Phase Plan

Goal: Build "Skill Smith" — an autonomous skill-authoring tool that turns a one-line project description (e.g. "AEM project with React", "Spring Boot REST + SOAP service with SQL") into a complete, tested set of Claude Agent Skills (SKILL.md folders) that make a coding agent an expert in that exact stack, with zero manual skill-hunting.

ENGINE: Skill Smith uses Claude Code itself as its engine — every research, generation, and verification step is performed by invoking the Claude Code CLI in headless mode (claude -p "<prompt>" --output-format stream-json or --output-format json, with --allowed-tools controlling permissions). The web UI is a thin orchestration + display layer over that engine. Claude Code calls are stateless per invocation; treat each stage's output as a file artifact on disk under workspace/<job>/ and pass prior artifacts explicitly into the next prompt.

GROUND TRUTH ABOUT SKILLS (the generated skills MUST follow this exactly): A skill is a directory containing a SKILL.md with YAML frontmatter (name, description required) plus a markdown body, and optional scripts/, references/, assets/ subfolders. Hard rules: (1) Progressive disclosure / three levels — name+description always in context (tiny); SKILL.md body loads when triggered (under ~500 lines); bundled files in references/ load only on demand; move long content into references/*.md and point to them. (2) The description is load-bearing — it is the ONLY trigger signal; write it to say both what the skill does AND specific contexts/phrases that should trigger it, slightly pushy to counter undertriggering; max ~1536 chars, keep well under. (3) One domain per skill, organized by variant — prefer a small family of focused skills over one mega-skill (e.g. for Spring Boot REST+SOAP+SQL: separate springboot-rest, springboot-soap, springboot-persistence-jpa). (4) Imperative voice, explain WHY a rule matters, include short Input->Output examples for non-obvious formats. (5) Code as tool vs doc — if a script should be run, say so and keep it out of context; if reference, put it in references/. (6) Lean bodies — link out, don't inline. (7) Safety — no malware/exfiltration; avoid baking in live URL fetches (supply-chain risk); bake researched knowledge in as static references. Where Anthropic's skill-creator exists on the machine (/mnt/skills/examples/skill-creator or via /plugin install skill-creator@anthropic-agent-skills), read and reuse its packaging/eval scripts (package_skill.py, run_loop.py) rather than reinventing them.

THE PIPELINE (discrete, observable stages; each stage is one or more claude -p calls; stream progress to the UI):
Stage 0 — Intake & clarification (interactive): user types a project description; run a scoping claude -p call decomposing it into target stack, distinct knowledge domains needed, likely agent tasks, and a list of open questions; render those questions in the UI as a short form (max ~5, prefer single/multi-select), e.g. for "AEM + React": AEM version (6.5/Cloud Service)? SPA Editor or traditional? Build tooling? TypeScript? Testing stack? Do NOT proceed to research until the user answers or says "use defaults".
Stage 1 — Intensive research: decompose answered scope into N research subtasks (one per domain); for each, invoke a Claude Code research agent with web access (claude -p "<research prompt>" --allowed-tools "WebSearch,WebFetch,Read,Bash"); each agent finds authoritative/primary sources, captures versioned facts, notes pitfalls and idioms, returns structured JSON brief { domain, key_apis, idioms, gotchas, version_notes, sources[] }; run in parallel where allowed, show live per-domain status; persist every brief to workspace/<job>/research/<domain>.json.
Stage 2 — Skill design: one claude -p call consumes all briefs and proposes a skill set plan (how many skills, each name, draft description, scope boundaries, which briefs feed it); enforce one-domain-per-skill/split-by-variant; show the plan in the UI for approve/edit before generation.
Stage 3 — Generation: for each planned skill a claude -p call writes the full directory: SKILL.md (frontmatter + lean body) plus references/*.md carrying heavy detail plus scripts/ where a deterministic helper beats prose; obey every GROUND TRUTH rule; write into workspace/<job>/skills/<name>/.
Stage 4 — Self-test before delivery (the differentiator, two kinds of checks): (a) Trigger reliability — generate 4-6 realistic prompts per skill that SHOULD trigger it (and 1-2 that should NOT); for each, ask a fresh claude -p agent (given only name+description among a list) whether it would load that skill; run each ~3x and compute a trigger rate; if mis/under-triggers, rewrite description and re-test (call skill-creator's run_loop.py if available). (b) Capability/knowledge check — spawn a fresh Claude Code agent WITH the skill loaded, give it a representative task from the stack, capture output, then a separate grader agent scores it against assertions derived from research briefs (correct APIs? right versions? known pitfalls avoided?); save report.json per skill; surface pass/fail in UI; on fail loop back to Stage 3 with grader feedback, cap iterations (e.g. 3).
Stage 5 — Package & deliver: validate frontmatter, body length, folder structure; package each skill (skill-creator package_skill.py if present, else zip into .skill); UI shows final skill set, each one's trigger rate + capability score, research sources, download links; emit install hints (~/.claude/skills/<name>/ personal or .claude/skills/<name>/ project).

THE BRAIN — how to call Claude Code: shell out to the claude CLI headless: claude -p "<prompt>" --output-format stream-json --allowed-tools "<tools>"; parse streamed JSON events and forward to the browser over WebSocket/SSE so the user watches research/generation live; make model and tool permissions configurable via env vars; default to the strongest available coding model for generation/grading; allow WebSearch/WebFetch ONLY in the research stage; cap parallelism and total invocations per job (cost guardrail) and show a running token/call estimate in the UI; fail gracefully — every stage retries with backoff, partial results always saved and downloadable.

THE UI — a clean web app (React + Node/Express or Python/FastAPI backend is fine): (1) Prompt bar "Describe your project" + submit. (2) Clarifier — Stage-0 questions as selectable chips/forms + "Use defaults" button. (3) Live pipeline view — vertical stepper Scope->Research->Design->Generate->Test->Package, each step streaming Claude Code output, per-domain research cards, approve/edit gates for the skill plan. (4) Results — skill set with per skill: trigger rate, capability score, expandable SKILL.md preview, sources list, download button; plus "Download all". (5) History — past jobs, re-runnable. Keep state on the backend keyed by job id; the browser is a view; persist everything to disk so a refresh never loses a job.

DELIVERABLES: a runnable repo (backend orchestrator + Claude Code invocation layer + WebSocket/SSE, frontend, README covering prerequisites: Node, the claude CLI installed and authenticated, env vars, npm run dev or equivalent); the five pipeline stages each independently testable; a workspace/ layout and an example completed job committed as a fixture; sensible defaults documented inline; a config file for model, tool perms, parallelism, and iteration caps.

BUILD ORDER: (1) Scaffold repo + the Claude Code invocation wrapper (stream parsing, retries) and prove it with a trivial claude -p "say hi" round-trip surfaced in the UI. (2) Stage 0 intake + clarifier end to end. (3) Stage 1 research with parallel agents + persisted briefs. (4) Stages 2-3 design + generation, validated against the GROUND TRUTH rules. (5) Stage 4 self-test (trigger + capability) with iterate-on-failure loop. (6) Stage 5 packaging + results screen. (7) Polish: history, cost meter, graceful failure, README.
